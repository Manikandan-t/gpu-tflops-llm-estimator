The One Big Idea You Must Understand

Before we go anywhere, here is the single most important concept in GPU selection for LLM inference:

💡 For most real LLM inference workloads, memory bandwidth matters more than compute TFLOPS.

A GPU with 100 TFLOPS and 4 TB/s bandwidth will usually outperform one with 200 TFLOPS and 2 TB/s bandwidth. This is counter-intuitive — we're trained to think "more compute equals more speed" — but it's wrong for the dominant workload in generative AI. The H200, for example, beats the H100 by roughly 1.9× on Llama-70B despite having identical compute. The only difference between them is memory: 4.8 TB/s versus 3.35 TB/s.

Why? Because generating each token requires reading the entire model from memory. The compute units finish their work in nanoseconds and then sit idle waiting for the next chunk of weights to arrive. We'll explain this in depth in the Fundamentals section with something called the roofline model — but keep this idea in your back pocket: for decode workloads, bandwidth is the real currency.

Part 1: What is a FLOP, really?

A FLOP stands for "Floating-point OPeration" — a single arithmetic operation on a floating-point number, like one multiplication or one addition. When we say a GPU delivers "1,000 TFLOPS," we mean it can perform one trillion (10¹²) of these operations every second.

The unit hierarchy you'll see repeatedly:

Unit conversions
1 KFLOPS = 10³ FLOPS (thousand)
1 MFLOPS = 10⁶ FLOPS (million)
1 GFLOPS = 10⁹ FLOPS (billion)
1 TFLOPS = 10¹² FLOPS (trillion) — the usual unit for GPUs
1 PFLOPS = 10¹⁵ FLOPS (quadrillion) — used for multi-GPU racks

Neural networks are essentially giant chains of multiplications and additions on floating-point numbers — matrix multiplications, dot products, activations. So FLOPS is a natural measure of raw computational capacity. The more FLOPS a GPU delivers, the more arithmetic it can do per second — in theory.

Part 2: How TFLOPS is calculated from hardware specs

Here's the formula every GPU datasheet is implicitly using:

The basic TFLOPS formula
TFLOPS = (Compute Units × Ops per Cycle × Clock GHz) / 1,000

Where:
• Compute Units = CUDA cores (for FP32) or Tensor Cores (for FP16/FP8/FP4)
• Ops per Cycle = how many operations each unit performs per clock tick
• Clock GHz = GPU frequency in billions of cycles per second
• / 1,000 = converts from GFLOPS to TFLOPS

Example 1: Calculating FP32 TFLOPS for H200 SXM

CUDA Cores: 16,896 (Hopper GH100 with one core disabled per SM)
Ops per Cycle: 2 (one Fused Multiply-Add = multiply + add = 2 ops)
Boost Clock: 1.98 GHz

FP32 = (16,896 × 2 × 1.98) / 1,000 = ~66.9 TFLOPS ✓
NVIDIA published: 67 TFLOPS ✓ (matches!)

Example 2: Why Tensor Cores multiply everything

Tensor Cores aren't scalar units. Each one is a specialized block that performs an entire matrix multiplication in a single cycle. A 4th-generation Hopper Tensor Core can do roughly 256 FP16 multiply-accumulate operations per cycle (512 ops total). That's why Tensor Core TFLOPS are orders of magnitude higher than CUDA core FP32:

H200 SXM FP16 Tensor Core (dense):
528 Tensor Cores × 256 FMA ops × 2 (multiply+add) × 1.98 GHz / 1,000
= ~535 TFLOPS dense
Double with 2:4 sparsity → ~1,070 TFLOPS sparse

NVIDIA published (sparse): 1,979 TFLOPS
NVIDIA published (dense): ~989.5 TFLOPS
⚠️ The sparse vs dense trap: NVIDIA's Hopper datasheets headline the sparse number (2× the dense number) assuming you're using a technique called 2:4 structured sparsity. Most real LLM workloads do NOT use this, so the dense number is what you actually get. We'll show both throughout this tool and label them clearly with DENSE and SPARSE badges.

Part 4: Memory bandwidth — the forgotten specification

Memory bandwidth measures how fast data can flow between the GPU's memory (HBM3e, GDDR7, etc.) and its compute units. It's measured in TB/s (terabytes per second). Here's why it matters:

Imagine your GPU has 1,000 TFLOPS of compute and 4 TB/s of bandwidth. In one second, it can theoretically do 1 trillion floating-point operations. But to do those operations, it needs data — and in one second, it can only load 4 terabytes of data from memory. If each operation consumes 2 bytes (an FP16 value), that's 2 trillion bytes needed for the 1T ops. We can only supply 4 trillion bytes — so in theory, compute should be able to keep up. But LLMs don't work this way: they load enormous weight matrices and use each value only briefly, so bandwidth is exhausted long before compute.

Part 5: Arithmetic intensity and the roofline model

The roofline model is the single most useful mental model in GPU performance analysis. It's a graph that tells you, for any given workload, whether you're limited by memory bandwidth or by compute.

The key metric is arithmetic intensity (AI):

Arithmetic Intensity
AI = FLOPs performed / Bytes transferred from memory

Units: FLOPs per Byte

And the hardware has a ridge point:

Hardware Ridge Point
AI_hw = Peak TFLOPS / Memory Bandwidth

If your workload's AI is below AI_hw → memory-bound (bandwidth limits you)
If your workload's AI is above AI_hw → compute-bound (TFLOPS limit you)

Calculating ridge points for real GPUs

What do these numbers mean? For an H100, your workload needs to perform at least 295 floating-point operations for every byte loaded from memory to fully utilize the GPU. If it does less than that, compute sits idle.

Where LLM inference falls on this graph

Here's the stunning insight: during single-request LLM decode (generating one token at a time for one user), arithmetic intensity is approximately 1 FLOP per byte. The ridge point is ~295. This means the H100 is idle approximately 99.7% of the time waiting for memory. Adding more TFLOPS does absolutely nothing to help.

This is why GPU selection for inference is fundamentally different from selection for training: training typically runs with large batch sizes that push arithmetic intensity up, making it compute-bound. Inference with small batches stays memory-bound.

🏭 The Factory Analogy: Imagine your GPU is a factory. The workers are Tensor Cores (very fast at assembly). The warehouse is HBM memory (holds all the parts = model weights). The delivery trucks are memory bandwidth (moves parts from warehouse to assembly line). In memory-bound mode (single-request decode), the workers finish their current job and then stand around waiting for the next truck to arrive. Adding more workers (more TFLOPS) doesn't help — they'll also stand around. You need faster trucks (more bandwidth). In compute-bound mode (prefill, or batch size 64+), trucks arrive faster than workers can process — now adding workers helps.

Part 6: The critical batch size — when memory-bound becomes compute-bound

For LLM decode, arithmetic intensity equals the batch size. Each weight value in memory is loaded once but used B times (once per request in the batch). So larger batches = higher AI = eventually compute-bound.

Critical Batch Size
B_crit ≈ Peak TFLOPS × bytes_per_parameter / Memory Bandwidth

For H100 SXM at BF16: B_crit ≈ 295
For H200 SXM at BF16: B_crit ≈ 206
For B200 at BF16: B_crit ≈ 281
For B200 at FP8: B_crit ≈ 562

Below B_crit = memory-bound. Above B_crit = compute-bound.

These batch sizes are extremely difficult to achieve in production. A serving system would need hundreds of concurrent requests, each at the same sequence position, to hit compute-bound operation for decode. That's why most LLM inference lives in the memory-bound regime, and why bandwidth dominates the conversation.

⚡ Throughput Math — The Formulas That Actually Work

In this section: The correct FLOPs-per-token formula (not the oversimplified one) • Memory-bound vs compute-bound throughput formulas • Real utilization numbers (MFU, MBU) • Worked examples

The FLOPs-per-Token Formula: What Really Happens

The v1.0 tool used a simplified formula: 2 × d² × num_layers. This is incomplete — it only captures a small piece of the Q and O attention projections. Let me show you the real breakdown.

The classical approximation (widely used)
C_forward ≈ 2 × P

Where P = non-embedding parameter count.
For Llama-70B: C ≈ 140 × 10⁹ FLOPs per token

This approximation works because every parameter participates in exactly one matrix multiplication per forward pass, and each multiply-accumulate takes 2 FLOPs (one multiply + one add).

The complete per-layer breakdown

Attention block per layer (standard MHA)
Q projection: 2 × d_model²
K projection: 2 × d_model²
V projection: 2 × d_model²
Attention scores QKᵀ: 2 × seq_len × d_model
Attention × V: 2 × seq_len × d_model
Output projection: 2 × d_model²
Subtotal: 8·d_model² + 4·seq_len·d_model
Feed-forward per layer (standard 4× expansion)
Two linear layers with d_ff = 4×d_model: 16 × d_model²
OR for SwiGLU (Llama-style): 6 × d_model × d_ff
Complete formula
C_forward = n_layers × [8·d_model² + 4·seq_len·d_model + 2·n_ffn·d_model·d_ff]
+ 2·d_model·vocab_size (unembedding)

Simplified: C_forward ≈ 2P + 2·n_layers·seq_len·d_attn
⚠️ When 2P breaks down: The simple approximation fails when seq_len > 8 × d_model. For Llama-70B (d_model = 8,192), that's ~65K tokens. At 128K context, attention FLOPs add ~15% overhead beyond 2P.
For MoE models like Mixtral or DeepSeek-V3, use C ≈ 2 × P_activated where P_activated = attention_params + k × per_expert_params. This is why Mixtral-8x7B actually has 47B total params but only 13B active — the FLOPs calculation uses 13B, not 47B.

Throughput Formulas for Each Regime

Memory-bound throughput (typical decode)

Memory-bound tokens/sec
tokens/sec = (B × MBU × Bandwidth) / (model_bytes + B × seq × KV_bytes_per_token)

Worked example: Llama-70B on H100, batch=1, BF16, MBU=80%
= (1 × 0.80 × 3,350 GB/s) / (140 GB)
= ~19 tok/s

Compute-bound throughput (prefill or large batch)

Compute-bound tokens/sec
tokens/sec = (MFU × Peak FLOPS) / (2P)

Worked example: Llama-70B prefill on H100, MFU=60%
= (0.60 × 989.5 × 10¹² FLOPS) / (140 × 10⁹ FLOPS/token)
= ~4,240 tok/s

Attention is always memory-bound during decode

This is a subtle but important point: even at large batch sizes, the attention portion of each layer stays memory-bound because each batch element has its own KV cache. So the arithmetic intensity of attention stays at ~1, regardless of batch size. Only the MLP portion transitions to compute-bound.

Step 1: Calculate Your Memory Budget

Before anything else, check if your model fits. Total GPU memory needed is:

Memory budget
Total = Model_Weights + KV_Cache + Activations + Overhead (~2GB)

Model_Weights = parameters × bytes_per_param
(FP16 = 2 bytes, FP8 = 1 byte, INT4 = 0.5 bytes)

KV_Cache (per token, per request) = 2 × n_layers × n_kv_heads × d_head × bytes_per_element
Total KV = per-token × seq_len × batch_size

Step 2: Recommended Configurations by Model Size

7-8B models

Single GPU is sufficient in all cases. A single H100 achieves ~12,500 tok/s batched. Even a consumer RTX 4090 handles this at ~70 tok/s for a single user. Production recommendation: 1× H100 or H200 for latency-sensitive serving; RTX PRO 6000 is fine for cost-sensitive workstation deployments.

13B models

Single GPU at FP16 on A100 80GB or any HBM GPU. H200 achieves ~12,000 tok/s with TensorRT-LLM batched. RTX 4090 works with INT4 quantization. Production recommendation: 1× H100/H200.

34B models

1× A100 80GB (FP8/INT8) or 1× H100. Two consumer GPUs needed with tensor parallelism at FP16. Single H200 (141 GB) is ideal and eliminates TP overhead. Production recommendation: 1× H200.

70B models — the critical inflection point

This is where hardware choices get interesting. FP16 requires 2× H100 minimum (TP=2). FP8 fits on a single H200 (141 GB) — this is why the H200 eliminates tensor parallelism overhead for 70B in MLPerf benchmarks. Real throughput: 1× H100 FP8 delivers ~460 tok/s at batch 64; 4× A6000 delivers 420-470 tok/s at much lower cost.

Production recommendation: 1× H200 FP8 (best single-GPU), or 2× H100 FP16 (if H200 unavailable), or 1× B200 (future-proof).

405B models

Multi-GPU mandatory. BF16 needs 810 GB — more than a single 8× H100 DGX node (640 GB total). FP8 (405 GB) fits on 8× H100 but leaves limited KV cache room. Best option: 8× H200 (1,128 GB total) or 8× B200 (1,536 GB total with FP4). DGX B200 with FP4 delivers >3× throughput versus DGX H200 on Llama 3.1 405B.

Production recommendation: 8× B200 or GB200 NVL72 rack.

Step 3: Parallelism Strategies

Tensor Parallelism (TP)

Splits weight matrices across GPUs horizontally, requiring AllReduce communication after every layer. NVLink is effectively mandatory: H100 SXM NVLink delivers 900 GB/s versus PCIe Gen4's 32 GB/s — a 28× difference. Scaling efficiency with NVLink: ~0.92× per GPU (2 GPUs = 1.84× throughput). With PCIe: 0.70-0.78× (2 GPUs = 1.4-1.5×). At TP=8, expect 25-44% overhead loss even with NVLink.

Pipeline Parallelism (PP)

Splits by layers vertically. Suitable for inter-node communication where bandwidth is lower. NVIDIA reports 1.2× speedup from adding PP in MLPerf Llama-70B benchmarks. Pipeline bubbles (idle time) are the main overhead.

Expert Parallelism (EP)

For MoE models, distributes experts across GPUs. vLLM's recommended mode for DeepSeek: TP=1, DP=8, EP enabled. Real gains vary: ~1.4% improvement on B200, up to ~11.1% on H200 for large MoE models. Usually combined with TP for dense layers and EP for sparse expert layers.

Step 4: Quantization Tradeoffs

Always try quantization before adding GPUs. Reducing precision from FP16 to INT4 cuts memory 4× and can avoid multi-GPU overhead entirely.

⚠️ Common Pitfalls and Misconceptions

Pitfall 1: "More TFLOPS = faster LLM inference"

A GPU with 100 TFLOPS and 4 TB/s bandwidth will outperform one with 200 TFLOPS and 2 TB/s for most inference workloads. During decode (which dominates wall-clock time for long outputs), the GPU computes one token at a time, reading the entire model from memory each step. The H200 outperforms H100 by ~1.9× on Llama-70B despite identical compute — purely due to bandwidth.

Pitfall 2: "Sparsity TFLOPS reflect real-world performance"

NVIDIA's Hopper datasheets quote "with sparsity" as the headline number, which is 2× the dense figure. Most models do not use 2:4 structured sparsity. Always use the dense number for LLM serving. An educational tool must clearly label whether numbers are dense or sparse, and default to dense for LLM workloads.

Pitfall 3: "Batch size doesn't matter for throughput planning"

The difference between batch=1 and batch=64 can be 64× in total throughput in the memory-bound regime, because weight-loading cost is amortized across all batch items. Any calculator that estimates throughput without asking about batch size and serving scenario will produce misleading results.

Pitfall 4: "Buy the GPU with the most memory"

Memory capacity is necessary but not sufficient. The RTX PRO 6000 has 96 GB GDDR7 — enough for a 70B FP8 model with room — but its 1.597 TB/s bandwidth (3× lower than H200) makes it severely memory-bound for LLM serving. You'll get the model to fit, but throughput will disappoint. For serving, prioritize bandwidth over capacity beyond what your model needs.

Pitfall 5: "Ignoring KV cache in capacity planning"

At 128K context, KV cache can exceed model weight size. Many teams size for "model fits in GPU" and then discover they can only serve one concurrent request because KV cache eats remaining memory. Always budget KV cache explicitly: KV_per_token × seq_len × batch_size.

Pitfall 6: "Cloud is always cheaper than on-prem"

Below 40% utilization, cloud wins. Above 85% utilization, on-prem wins by 5-10×. Most teams fall somewhere in between and should run hybrid. The math changes dramatically with GPU generation too: H100 cloud is now 3× cheaper than 2023, so older on-prem analyses may have stale conclusions.

Pitfall 7: "Quantization is always faster"

Only when paired with optimized kernels. AWQ without Marlin runs at 67 tok/s; with Marlin, 741 tok/s. The kernel matters more than the method. If your serving stack doesn't support Marlin or similar, you may get worse performance from quantized models than FP16.

The common thread in all these pitfalls: headline specs don't match real-world performance. Always benchmark on your actual workload before committing to hardware. Every claim in this tool is supported by primary sources, but your specific use case may still behave differently.
